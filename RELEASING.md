# Release Cycle

This document defines the release cycle, versioning, and publishing process for the Monet platform. We intentionally use a lightweight, "just enough process" approach to keep releases fast and developer-friendly.

## Versioning Scheme

Monet follows [Semantic Versioning](https://semver.org/) (`vX.Y.Z`), where:

- **Major (`vX.0.0`)**: Incompatible API changes, major breaking changes to the database schema requiring complex manual migrations, or significant architectural shifts.
- **Minor (`v0.Y.0` or `vX.Y.0`)**: Backwards-compatible new features, new API endpoints, or significant enhancements.
- **Patch (`vX.Y.Z`)**: Backwards-compatible bug fixes, security patches, or minor internal refactoring.

> **Note**: While Monet is in early development (`0.x.x` series), minor versions may contain breaking changes as the API stabilizes.

## What is a Release?

A **Release** is a snapshot of the `main` branch that has been explicitly tagged, verified, and published as a set of Docker images to the GitHub Container Registry (GHCR) alongside a GitHub Release.

Published release artifacts:
- `ghcr.io/team-monet/monet-api`
- `ghcr.io/team-monet/monet-dashboard`
- `ghcr.io/team-monet/monet-migrate`

Tag behavior:
- Every release tag publishes matching image tags (for example, `v0.1.0`).
- Stable tags (no prerelease suffix) also update `latest`.
- Prerelease tags containing `-` (for example, `v0.2.0-rc.1`) do **not** update `latest`.

Regular merges to the `main` branch are continuously integrated and tested, but they are not automatically published as releases. Releases are intentional cut points intended for production or self-hosted deployments.

## Release Trigger

Currently, releases are triggered manually via **Git tags**.

To trigger a release:
```bash
# 1. Ensure you are on the latest main branch
git checkout main
git pull origin main

# 2. Create an annotated tag for the new version
git tag -a v0.1.0 -m "Release v0.1.0"

# 3. Push the tag to GitHub
git push origin v0.1.0
```

*Note: Release tags must point to a commit already merged to `main`. Pushing the tag triggers CI/CD to build and publish the Docker images and create a **draft** GitHub Release for maintainer review.*

## Changelog Generation

We avoid manual changelog maintenance. Changelogs are **auto-generated by GitHub Releases** based on merged Pull Request titles since the last tag. 

To ensure high-quality changelogs:
- Use clear, descriptive PR titles.
- Optionally follow [Conventional Commits](https://www.conventionalcommits.org/) for PR titles (e.g., `feat: add memory TTL`, `fix: resolve MCP connection timeout`).
- When drafting the final release on GitHub, the maintainer can organize or edit the auto-generated notes to highlight the most important changes.

## Release Cadence

- **Current (Early Stage)**: **Ad-hoc**. Releases are cut whenever a significant feature is completed, a critical bug is fixed, or the team feels a new snapshot is ready for users.
- **Future (Maturation)**: As the project stabilizes and adoption grows, we will tighten the cadence to a scheduled release cycle (e.g., bi-weekly or monthly), with patch releases cut as needed for hotfixes.

## Pre-Release Checklist

Before tagging and pushing a new release, verify the following:

- [ ] **CI is green**: Ensure the latest commit on `main` has passed all CI checks (unit tests, integration tests, linting).
- [ ] **Local verification**: Run `pnpm local:up` and verify the system starts cleanly without errors.
- [ ] **Documentation**: Ensure `README.md` and the `docs/` directory reflect any new features, configuration changes, or upgrade steps.
- [ ] **Migrations**: Verify that any new database migrations apply cleanly to an existing state (not just fresh databases).
- [ ] **Version bumps (if any)**: Ensure any hardcoded versions in `docker-compose.runtime.yml` or sample `.env` files are updated if required.

## Post-Release

Once the GitHub Action completes:
1. Go to the [Releases page](https://github.com/team-monet/monet/releases).
2. Open and edit the newly drafted release.
3. Review the auto-generated changelog, highlight key features, and mention any breaking changes or required migration steps.
4. Publish the release when review is complete.
